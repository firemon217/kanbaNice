## 2. НАЗНАЧЕНИЕ И ЦЕЛИ

**Назначение системы:** Автоматизация управления проектами и задачами в командах малого и среднего размера методом Kanban.

**Цели разработки:**
1. Предоставить веб-интерфейс для управления задачами в браузере
2. Предоставить десктопное приложение как альтернативный клиент
3. Реализовать безопасную аутентификацию и авторизацию
4. Обеспечить масштабируемое REST API для возможной интеграции

---

## 3. ТРЕБОВАНИЯ К СИСТЕМЕ

### 3.1. Функциональные требования

#### 3.1.1. Подсистема аутентификации
- **ФТ-01:** Система должна поддерживать регистрацию по email/пароль
- **ФТ-02:** Система должна поддерживать вход через Google (OAuth2)
- **ФТ-03:** Система должна обеспечивать восстановление пароля по email
- **ФТ-04:** Все защищённые маршруты требуют валидного JWT-токена

#### 3.1.2. Подсистема компаний и пользователей
- **ФТ-05:** Администратор может создать компанию и управлять её данными
- **ФТ-06:** Администратор может добавлять и удалять сотрудников компании
- **ФТ-07:** Пользователь может редактировать свой профиль и удалить аккаунт

#### 3.1.3. Подсистема проектов
- **ФТ-08:** Администратор создаёт проекты в рамках компании
- **ФТ-09:** Администратор добавляет участников в проект с ролью WORKER
- **ФТ-10:** Система поддерживает CRUD-операции для проектов

#### 3.1.4. Подсистема Kanban
- **ФТ-11:** В проекте можно создавать неограниченное количество досок
- **ФТ-12:** На доске можно создавать задачи с заголовком и описанием
- **ФТ-13:** Задача имеет статус TODO или DONE
- **ФТ-14:** Система поддерживает CRUD-операции для досок и задач

### 3.2. Требования к архитектуре (PCMEF)

- **АТ-01:** Архитектура реализует паттерн PCMEF (P-C-M-E-F)
- **АТ-02:** Зависимости направлены строго: P→C→M→E→F
- **АТ-03:** Слои взаимодействуют через интерфейсы
- **АТ-04:** Бизнес-логика изолирована в слое Mediator (service/)
- **АТ-05:** Доступ к БД только через слой Foundation (Repository/)

### 3.3. Требования к безопасности
- **БТ-01:** Пароли хранятся в виде BCrypt-хешей
- **БТ-02:** Аутентификация через JWT-токены с подписью
- **БТ-03:** Поддержка OAuth2 (Google) без хранения паролей провайдера
- **БТ-04:** HTTPS через Nginx SSL-терминацию
- **БТ-05:** CORS-политика настраивается через WebConfig

### 3.4. Требования к интерфейсам

#### Веб-клиент (React)
- **ИТ-01:** Не менее 5 страниц (выполнено: 7)
- **ИТ-02:** Адаптивная вёрстка
- **ИТ-03:** SPA с клиентской маршрутизацией (React Router)

#### Десктоп-клиент (JavaFX)
- **ИТ-04:** Не менее 5 экранов (выполнено: 7)
- **ИТ-05:** Нативный GUI на JavaFX 21

### 3.5. Требования к REST API
- **АПИ-01:** Не менее 15 эндпоинтов (выполнено: 27)
- **АПИ-02:** Документация OpenAPI (Swagger UI)
- **АПИ-03:** Стандартные HTTP-статусы (200, 201, 400, 401, 403, 404, 500)
- **АПИ-04:** Валидация входных данных через @Valid

### 3.6. Требования к развёртыванию
- **РТ-01:** Docker + Docker Compose для всех компонентов
- **РТ-02:** Nginx как реверс-прокси
- **РТ-03:** PostgreSQL 16 в качестве СУБД

### 3.7. Нефункциональные требования
| Требование | Значение |
|-----------|---------|
| Время отклика API | < 500 мс при нагрузке до 100 RPS |
| Доступность | 99% (разработка/учёба) |
| Масштабируемость | Горизонтальная через Docker |
| Поддерживаемость | Код соответствует принципам SOLID |
