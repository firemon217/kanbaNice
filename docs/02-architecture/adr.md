# Архитектурные решения (ADR)

## ADR-001: Выбор архитектурного паттерна PCMEF

**Статус:** Принято  
**Дата:** 07.05.2026

### Контекст
Необходимо выбрать архитектурный паттерн для организации кода серверной части, обеспечивающий разделение ответственности и тестируемость каждого слоя.

### Решение
Использовать PCMEF (Presentation–Control–Mediator–Entity–Foundation), маппированный на слои Spring Boot:
- **P** → React (web) + JavaFX (desktop)
- **C** → `controller/` (Spring @RestController)
- **M** → `service/` (Spring @Service)
- **E** → `entity/` (JPA @Entity)
- **F** → `Repository/` (Spring Data JPA)

### Последствия
- Контроллеры не содержат бизнес-логики
- Сервисы не знают об HTTP-протоколе
- Репозитории содержат только SQL-запросы через JPA
- Возможно независимое тестирование каждого слоя

## ADR-002: Аутентификация через JWT + OAuth2

**Статус:** Принято  
**Дата:** 07.05.2026

### Контекст
Система должна поддерживать два способа входа: традиционный (email/пароль) и через Google. Оба клиента (web и desktop) должны использовать единый механизм.

### Рассмотренные варианты
1. Session-based (HttpSession) — не подходит для desktop-клиента
2. JWT — stateless, работает для любого клиента
3. JWT + OAuth2 — покрывает оба способа входа

### Решение
JWT (JJWT 0.12.6) + Spring Security OAuth2. После OAuth2-входа генерируется JWT, одинаково используемый web и desktop.

### Последствия
- Сервер stateless — нет хранения сессий
- Desktop-клиент отправляет `Authorization: Bearer <token>`
- Refresh-токены не реализованы (принято как компромисс)

## ADR-003: Многоконтейнерная архитектура (Docker Compose)

**Статус:** Принято  
**Дата:** 10.05.2026

### Контекст
Система состоит из четырёх компонентов: PostgreSQL, Spring Boot backend, React frontend, Nginx. Нужен воспроизводимый способ развёртывания.

### Решение
Docker Compose с четырьмя сервисами:
- `postgres` — база данных с persistent volume
- `backend` — Spring Boot (port 8080)
- `frontend` — React/Vite (port 5173, production build)
- `nginx` — реверс-прокси (ports 80/443, SSL)

### Последствия
- Однокомандное развёртывание: `docker-compose up --build`
- Зависимости между сервисами через `depends_on` + `healthcheck`
- SSL-терминация на уровне Nginx

## ADR-004: Два типа клиентов на одном API

**Статус:** Принято  
**Дата:** 12.05.2026

### Контекст
МУ требует Траекторию Г (Enterprise): Desktop UI + Web UI с единой бизнес-логикой.

### Решение
- Web: React SPA (Vite, Axios) — обращается к REST API через HTTPS
- Desktop: JavaFX + Jackson — обращается к тому же REST API через ApiClient
- Единственный источник данных — Spring Boot backend

### Последствия
- Полная независимость клиентов: изменения в одном не затрагивают другой
- Одинаковая бизнес-логика для обоих интерфейсов
- Десктоп работает без браузера

## ADR-005: ModelMapper для маппинга Entity ↔ DTO

**Статус:** Принято  
**Дата:** 15.05.2026

### Контекст
Необходимо преобразовывать JPA-сущности в DTO и обратно без дублирования кода.

### Рассмотренные варианты
1. Ручной маппинг — надёжно, но избыточно
2. MapStruct — compile-time, высокая производительность
3. ModelMapper — runtime, минимум конфигурации

### Решение
ModelMapper 3.2.0: автоматический маппинг по именам полей.

### Последствия
- Минимальный код маппинга в сервисах
- Слой Entity изолирован от API-контрактов (DTO)
- Реализует паттерн Data Mapper (Этап 6)
